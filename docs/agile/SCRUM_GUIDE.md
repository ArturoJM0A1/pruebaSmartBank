# Scrum Guide - SmartBank

## Overview

Scrum is an agile framework for developing complex products. It's iterative, incremental, and focuses on delivering value quickly.

---

## Scrum Roles

### Product Owner (PO)
**Responsibility**: Maximize product value

- Manages product backlog
- Defines user stories and acceptance criteria
- Prioritizes work based on business value
- Accepts or rejects completed work
- Represents stakeholders

**Key Skills**: Business knowledge, communication, decision-making

### Scrum Master (SM)
**Responsibility**: Ensure Scrum is understood and enacted

- Coaches the team on Scrum practices
- Removes impediments (blockers)
- Facilitates Scrum events
- Protects the team from outside distractions
- Fosters continuous improvement

**Key Skills**: Servant leadership, facilitation, coaching

### Development Team
**Responsibility**: Deliver a "Done" increment each sprint

- Self-organizing (team decides how to do the work)
- Cross-functional (has all skills needed to deliver)
- 3-9 members (optimal size)
- Collective ownership (everyone is responsible)

**Key Skills**: Technical skills, collaboration, accountability

---

## Scrum Events

### Sprint
**Duration**: 1-4 weeks (2 weeks for SmartBank)
**Purpose**: Create a "Done" increment

The Sprint is the heart of Scrum. All other events happen within it.

### Sprint Planning
**Duration**: 2-4 hours for 2-week sprint
**Purpose**: Define what can be done and how

- **What**: Product Owner presents prioritized backlog items
- **How**: Development Team breaks down stories into tasks
- **Capacity**: Team estimates how much work they can handle

### Daily Scrum (Standup)
**Duration**: 15 minutes maximum
**Purpose**: Synchronize work and identify blockers

Three questions for each team member:
1. What did I accomplish yesterday?
2. What will I work on today?
3. Are there any blockers?

### Sprint Review
**Duration**: 1-2 hours
**Purpose**: Inspect the increment and adapt the backlog

- Demo completed work to stakeholders
- Gather feedback
- Update product backlog based on feedback
- Discuss what was not completed and why

### Sprint Retrospective
**Duration**: 1-1.5 hours
**Purpose**: Reflect on the process and create improvements

- What went well?
- What could improve?
- Create action items for next sprint

## Scrum Artifacts

### Product Backlog
**What**: Ordered list of everything needed in the product
**Owner**: Product Owner
**Characteristics**:
- Dynamic (changes as learning occurs)
- Never complete (always evolving)
- Single source of work for the team

### Sprint Backlog
**What**: Items selected for the current sprint + plan
**Owner**: Development Team
**Characteristics**:
- Real-time picture of work
- Updated daily
- Visible to all

### Increment
**What**: The sum of completed backlog items at sprint end
**Owner**: Development Team
**Characteristics**:
- Must be "Done" (meets Definition of Done)
- Usable (potentially releasable)
- Adds value

### Definition of Done (DoD)
A shared understanding of what "done" means:

- Code is written and follows style guide
- Unit tests are written and passing
- Code is reviewed
- Documentation is updated
- No critical bugs
- Product Owner accepted

## Scrum vs Kanban

| Aspect | Scrum | Kanban |
|--------|-------|--------|
| Cadence | Fixed sprints | Continuous flow |
| Roles | PO, SM, Team | No prescribed roles |
| Changes | Not during sprint | Any time |
| Metrics | Velocity | Lead time, cycle time |
| Best for | Product development | Operations, support |

### When to Use Each

**Use Scrum when**:
- Building a new product
- Need regular checkpoints
- Team is new to agile
- Work can be grouped into sprints

**Use Kanban when**:
- Work is unpredictable
- Need to respond quickly to changes
- Team is experienced
- Focus on flow efficiency

## Scrum Values

1. **Courage**: Do the right thing, face tough problems
2. **Focus**: Focus on Sprint goals
3. **Openness**: Be transparent about work and challenges
4. **Respect**: Value each other's skills and perspectives
5. **Commitment**: Commit to achieving Sprint goals

---

## Common Scrum Anti-Patterns

1. **Scrum-but**: "We do Scrum, but..."
2. **Zombie Scrum**: Going through motions without value
3. **Scrum-fall**: Waterfall disguised as Scrum
4. **Scrum master as manager**: SM should be servant leader
5. **Ignoring retrospectives**: Missed improvement opportunities
